Strictly use Conventional Commits.
Categories:

- 'feat' for new features
- 'fix' for bug fixes
- 'docs' for documentation
- 'style' for formatting
- 'refactor' for code changes that do not add features
- 'chore' for maintenance tasks.

If there are new lines of logic or new source code files, always use 'feat'.
If the diff removes errors or modifies existing faulty logic, always use 'fix'.
If changes are only on style CSS, whitespace, or semicolons, always use 'style'.
If changes are only on variable names or code cleanup without changing functionality, always use 'refactor'.

If 100% of the changes are in .md files or comments, always use 'docs'.
If the change affects config files (as package.json, .gitignore, _.json, _.yaml, _.yml, _.xml, _.toml, _.ini, _.env, _.env.example), always use 'chore'.

Provide a final structured text for the commit message with the description of the Commit in Conventional Commits format. Ignore the JIRA_ID parameter if there is no explicit JIRA_ID. You must use the following commit structure:

{CATEGORY}({SCOPE}): {JIRA_ID} {DESCRIPTION_SUMMARY}

✅ {BUSINESS_CONTEXT_AND_RATIONALE_FOR_THE_CHANGE}
Changes:
✅ {CHANGE1}
✅ {CHANGE2}
Impact:
✅ {IMPACT1}
✅ {IMPACT2}
Docs:
✅ {LINK_TO_RELATIVE_PATH_OF_STORY_MD_FILE_IF_EXISTS}

"
